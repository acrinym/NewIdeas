## CYC-2026-027: ICO/CUR Format Swapping (Type Field Confusion)

**Severity:** MEDIUM  
**Component:** Cursor loading, icon loading, theme/skin assets  
**Status:** 🔍 **INVESTIGATING**

### Background: ICO and CUR Are Nearly Identical

Both formats use the same structure with **one byte difference:**

```
Offset  ICO File              CUR File
------  --------------------  --------------------
0x00    00 00 (Reserved)      00 00 (Reserved)
0x02    01 00 (Type=ICO)      02 00 (Type=CUR)    ← Only difference!
0x04    Count (# of images)   Count (# of images)
0x06+   Directory entries     Directory entries
        Image data            Image data
```

**Directory entry differences:**

```
Offset  ICO Entry             CUR Entry
------  --------------------  --------------------
0x00    Width                 Width
0x01    Height                Height
0x02    ColorCount            ColorCount
0x03    Reserved              Reserved
0x04    Color Planes          Hotspot X           ← Different semantics
0x06    Bits Per Pixel        Hotspot Y           ← Different semantics
0x08    Image Size            Image Size
0x0C    Image Offset          Image Offset
```

**This means you can convert ICO ↔ CUR by changing ONE BYTE.**

### The Attack

#### Attack Vector 1: Load ICO as Cursor

Create an ICO file with malicious image data. Change byte 0x02 from `01` to `02`. Now it's a "cursor."

**Cursor loader expects:**
- Hotspot X/Y coordinates at offset 0x04-0x07
- But those bytes contain Color Planes and Bits Per Pixel (from ICO)

**What happens:**
- Hotspot coordinates are garbage (interpreted as color/bpp values)
- Cursor displays at wrong position
- **OR: Hotspot values cause integer overflow/buffer issues in renderer**

**Example:**
```
ICO file has:
  Offset 0x04: Color Planes = 0x0001
  Offset 0x06: Bits Per Pixel = 0x0020 (32 bpp)

Loaded as CUR:
  Hotspot X = 0x0001 (fine)
  Hotspot Y = 0x0020 (fine, but unexpected)

If cursor renderer doesn't validate hotspot:
  Hotspot X = 0xFFFF (from malicious ICO)
  Hotspot Y = 0xFFFF (from malicious ICO)
  → Render cursor 65535 pixels away from click point
  → Potential out-of-bounds access in rendering code
```

#### Attack Vector 2: Load CUR as Icon

Create a CUR file. Change byte 0x02 from `02` to `01`. Now it's an "icon."

**Icon loader expects:**
- Color Planes and Bits Per Pixel at offset 0x04-0x07
- But those bytes contain Hotspot X/Y (from CUR)

**What happens:**
- Color format is misinterpreted
- Image decoder receives wrong parameters
- **OR: Invalid bpp value causes decoder crash**

**Example:**
```
CUR file has:
  Offset 0x04: Hotspot X = 0x0010 (16 pixels)
  Offset 0x06: Hotspot Y = 0x0010 (16 pixels)

Loaded as ICO:
  Color Planes = 0x0010 (16 planes - INVALID!)
  Bits Per Pixel = 0x0010 (16 bpp - unusual but valid)

Decoder attempts to process 16 color planes:
  → Most decoders expect 1 plane
  → Confusion, crash, or wrong rendering
```

#### Attack Vector 3: Polyglot ICO/CUR

**A single file that behaves differently based on how it's loaded:**

```python
# Create polyglot.ico:
with open('polyglot.ico', 'wb') as f:
    f.write(b'\x00\x00')  # Reserved
    f.write(b'\x01\x00')  # Type = ICO
    f.write(b'\x01\x00')  # Count = 1 image
    
    # Directory entry crafted to be "safe" as ICO but malicious as CUR
    f.write(b'\x20\x20')  # Width=32, Height=32
    f.write(b'\x00\x00')  # ColorCount=0, Reserved=0
    f.write(b'\xFF\xFF')  # Planes=65535 OR HotspotX=65535
    f.write(b'\xFF\xFF')  # BPP=65535 OR HotspotY=65535
    f.write(struct.pack('<I', image_size))
    f.write(struct.pack('<I', image_offset))
    
    # Image data (malicious PNG or BMP)
    f.write(malicious_image_data)
```

**As ICO:** Invalid (65535 planes), might crash icon loader  
**As CUR:** Invalid hotspot (65535, 65535), might crash cursor renderer  
**Purpose:** Test if either loader has insufficient validation

### Why This Matters

**Windows and most parsers rely on the Type field:**
- Byte 0x02 = `01` → Load as ICO
- Byte 0x02 = `02` → Load as CUR

**But if any code ignores the Type field** and tries to be "smart" about detecting format:
- Looks at file extension (`.ico` vs `.cur`)
- Assumes extension matches type
- Doesn't validate Type field

**An attacker can:**
1. Name a CUR file as `evil.ico`
2. Windows loads it as ICO (by extension)
3. Cycloside loads it as ICO (trusting extension)
4. But it contains CUR-specific data
5. Decoder confusion → crash or exploitation

### Historical Precedent

**CVE-2010-3970: Windows Graphics Component (ICO/CUR parsing)**
- Improper validation of ICO/CUR directory entries
- Malformed icon caused buffer overflow
- Remote code execution

**This proves ICO/CUR parsing is exploitable if not careful.**

### Mitigation

```csharp
private static bool ValidateIcoFile(string path)
{
    var bytes = File.ReadAllBytes(path);
    if (bytes.Length < 6) return false;
    
    // Check Reserved field
    if (bytes[0] != 0x00 || bytes[1] != 0x00) return false;
    
    // Check Type field matches expected format
    ushort type = BitConverter.ToUInt16(bytes, 2);
    
    // For ICO files, Type MUST be 0x0001
    if (type != 0x0001)
    {
        Logger.Log($"ICO file has wrong type field: {type:X4}");
        return false;
    }
    
    // Validate directory entries
    ushort count = BitConverter.ToUInt16(bytes, 4);
    for (int i = 0; i < count; i++)
    {
        int entryOffset = 6 + (i * 16);
        if (entryOffset + 16 > bytes.Length) return false;
        
        // Read color planes and bpp
        ushort planes = BitConverter.ToUInt16(bytes, entryOffset + 4);
        ushort bpp = BitConverter.ToUInt16(bytes, entryOffset + 6);
        
        // Validate planes (should be 0 or 1)
        if (planes > 1)
        {
            Logger.Log($"ICO has invalid color planes: {planes}");
            return false;
        }
        
        // Validate bpp (common values: 1, 4, 8, 16, 24, 32)
        if (bpp != 1 && bpp != 4 && bpp != 8 && bpp != 16 && 
            bpp != 24 && bpp != 32 && bpp != 0)
        {
            Logger.Log($"ICO has unusual bpp: {bpp}");
            return false;
        }
    }
    
    return true;
}

private static bool ValidateCurFile(string path)
{
    var bytes = File.ReadAllBytes(path);
    if (bytes.Length < 6) return false;
    
    // Check Reserved field
    if (bytes[0] != 0x00 || bytes[1] != 0x00) return false;
    
    // Check Type field matches expected format
    ushort type = BitConverter.ToUInt16(bytes, 2);
    
    // For CUR files, Type MUST be 0x0002
    if (type != 0x0002)
    {
        Logger.Log($"CUR file has wrong type field: {type:X4}");
        return false;
    }
    
    // Validate directory entries
    ushort count = BitConverter.ToUInt16(bytes, 4);
    for (int i = 0; i < count; i++)
    {
        int entryOffset = 6 + (i * 16);
        if (entryOffset + 16 > bytes.Length) return false;
        
        // Read hotspot coordinates
        ushort hotspotX = BitConverter.ToUInt16(bytes, entryOffset + 4);
        ushort hotspotY = BitConverter.ToUInt16(bytes, entryOffset + 6);
        
        // Read image dimensions
        byte width = bytes[entryOffset];
        byte height = bytes[entryOffset + 1];
        
        // Hotspot must be within image bounds
        if (hotspotX > (width == 0 ? 256 : width))
        {
            Logger.Log($"CUR hotspot X out of bounds: {hotspotX}");
            return false;
        }
        if (hotspotY > (height == 0 ? 256 : height))
        {
            Logger.Log($"CUR hotspot Y out of bounds: {hotspotY}");
            return false;
        }
    }
    
    return true;
}

// Load with explicit type checking
private static Cursor LoadCursor(string path)
{
    // Validate as CUR before loading
    if (!ValidateCurFile(path))
        throw new InvalidDataException("Invalid cursor file");
    
    // Additional check: Reject if file claims to be ICO
    var bytes = File.ReadAllBytes(path);
    ushort type = BitConverter.ToUInt16(bytes, 2);
    if (type != 0x0002)
        throw new InvalidDataException("File is not a cursor (wrong type field)");
    
    return new Cursor(path);
}
```

**Key principles:**
- **Validate Type field explicitly** (don't trust extension)
- **Reject files with wrong Type** (ICO claiming to be CUR or vice versa)
- **Validate format-specific fields** (hotspot for CUR, planes/bpp for ICO)
- **Don't rely on "smart" format detection** (check Type field first)

## CYC-2026-028: WAV Decoder Exploits via Malformed Audio Data

**Severity:** MEDIUM (historically low frequency, but impact is HIGH if exploited)  
**Component:** AudioService (NAudio), WAV decoding  
**Status:** 🔍 **INVESTIGATING**

### Background: WAV Files Have Been Exploited

**Yes, WAV exploits exist, though they're less common than image format exploits:**

#### Historical WAV Vulnerabilities

**CVE-2018-0802: Microsoft Office Equation Editor**
- Malformed WAV file embedded in Office document
- Integer overflow in audio decoder
- Remote code execution

**CVE-2006-2916: libsndfile (used by many audio players)**
- Heap overflow in WAV parsing
- Malformed `fmt` chunk caused buffer overflow
- Arbitrary code execution

**CVE-2017-0558: Windows Media Foundation**
- WAV file with crafted `data` chunk
- Out-of-bounds read
- Information disclosure / crash

**So yes, WAV exploits are real, just less common than PNG/JPEG exploits.**

### Why WAV Exploits Are Less Common

1. **Simpler format** than PNG/JPEG (less compression, less complexity)
2. **Less attack surface** (no Huffman tables, no complex transforms)
3. **Less scrutiny** (attackers focus on image formats that browsers display automatically)
4. **Native decoders** are often simpler (PCM audio is straightforward)

**But simpler doesn't mean safe.**

### Attack Vectors in WAV Files

#### Attack Vector 1: Malformed `fmt` Chunk

```
fmt chunk structure:
  ChunkID: "fmt " (4 bytes)
  ChunkSize: 16 or 18 or 40 (4 bytes)
  AudioFormat: 1=PCM, 3=Float, etc. (2 bytes)
  NumChannels: 1=Mono, 2=Stereo (2 bytes)
  SampleRate: e.g. 44100 Hz (4 bytes)
  ByteRate: SampleRate * NumChannels * BitsPerSample/8 (4 bytes)
  BlockAlign: NumChannels * BitsPerSample/8 (2 bytes)
  BitsPerSample: 8, 16, 24, 32 (2 bytes)
```

**Malicious fmt chunk:**
```
AudioFormat: 1 (PCM)
NumChannels: 65535 (invalid - should be 1 or 2)
SampleRate: 4294967295 (4GB Hz - impossible)
BitsPerSample: 32768 (invalid - should be 8/16/24/32)
```

**Decoder calculates buffer size:**
```csharp
int bufferSize = NumChannels * SampleRate * BitsPerSample / 8;
// = 65535 * 4294967295 * 32768 / 8
// = Integer overflow → wraps to small value
// Decoder allocates small buffer
// Writes large amount of data
// Buffer overflow!
```

#### Attack Vector 2: Metadata Chunk Exploits

**WAV files can contain metadata chunks:**
- `LIST` - List of metadata (artist, title, etc.)
- `INFO` - Information chunk
- `bext` - Broadcast extension
- Custom chunks (any 4-character ID)

**Malicious metadata:**
```
LIST chunk:
  ChunkID: "LIST" (4 bytes)
  ChunkSize: 0xFFFFFFFF (4 bytes) ← Lies about size
  Type: "INFO" (4 bytes)
  [Malicious data claiming to be 4GB]
```

**If NAudio or any viewer processes metadata without size validation:**
- Buffer overflow
- Integer overflow
- Denial of service

#### Attack Vector 3: Multi-Channel Audio Confusion

**NAudio supports multi-channel audio:**
- Stereo (2 channels)
- 5.1 Surround (6 channels)
- 7.1 Surround (8 channels)

**What if NumChannels = 255?**
```csharp
float[] buffer = new float[SampleCount * NumChannels];
// If SampleCount = 100000, NumChannels = 255
// buffer size = 25,500,000 floats = 102 MB (for one buffer)
// If decoder creates multiple buffers... memory exhaustion
```

**This is less severe than chunk size overflow (CYC-2026-026) but still a DoS vector.**

#### Attack Vector 4: Compressed Audio Formats

**WAV can contain compressed audio:**
- AudioFormat = 2: ADPCM
- AudioFormat = 6: A-law
- AudioFormat = 7: μ-law
- AudioFormat = 0x0055: MP3 (yes, MP3 in WAV!)

**Compressed formats have decoders with historical vulnerabilities.**

**If NAudio passes compressed WAV to native decoder:**
- Malformed ADPCM could trigger buffer overflow in decompressor
- Malformed MP3-in-WAV could exploit MP3 decoder bugs

### Current Protections in Cycloside

**From CYC-2026-006:**
```csharp
private static readonly long MaxAudioFileSize = 10 * 1024 * 1024; // 10 MB

var fileInfo = new FileInfo(path);
if (fileInfo.Length > MaxAudioFileSize) return;
```

**This prevents:**
- Large files from being loaded (good)

**This DOESN'T prevent:**
- Malformed `fmt` chunk causing integer overflow
- Metadata chunk exploits
- Multi-channel memory exhaustion
- Compressed format decoder exploits

### Mitigation

**Add WAV structure validation (from CYC-2026-026 expanded):**

```csharp
private static bool ValidateWavFile(string path)
{
    using var stream = File.OpenRead(path);
    using var reader = new BinaryReader(stream);
    
    // RIFF header
    var riff = new string(reader.ReadChars(4));
    if (riff != "RIFF") return false;
    
    uint fileSize = reader.ReadUInt32();
    if (fileSize > stream.Length || fileSize < stream.Length - 8)
        return false;
    
    var wave = new string(reader.ReadChars(4));
    if (wave != "WAVE") return false;
    
    bool foundFmt = false;
    bool foundData = false;
    
    // Process chunks
    while (stream.Position < stream.Length - 8)
    {
        var chunkId = new string(reader.ReadChars(4));
        uint chunkSize = reader.ReadUInt32();
        
        // Validate chunk size
        long remaining = stream.Length - stream.Position;
        if (chunkSize > remaining)
        {
            Logger.Log($"WAV chunk size exceeds file: {chunkId}");
            return false;
        }
        
        // Validate fmt chunk specifically
        if (chunkId == "fmt ")
        {
            foundFmt = true;
            
            // Read fmt data
            ushort audioFormat = reader.ReadUInt16();
            ushort numChannels = reader.ReadUInt16();
            uint sampleRate = reader.ReadUInt32();
            uint byteRate = reader.ReadUInt32();
            ushort blockAlign = reader.ReadUInt16();
            ushort bitsPerSample = reader.ReadUInt16();
            
            // Validate audio format
            if (audioFormat == 1) // PCM
            {
                // OK - uncompressed
            }
            else if (audioFormat == 3) // Float
            {
                // OK - float PCM
            }
            else
            {
                Logger.Log($"Unsupported WAV format: {audioFormat}");
                return false; // Reject compressed formats
            }
            
            // Validate channels (1=mono, 2=stereo, 6=5.1, 8=7.1)
            if (numChannels < 1 || numChannels > 8)
            {
                Logger.Log($"Invalid channel count: {numChannels}");
                return false;
            }
            
            // Validate sample rate (8000 to 192000 Hz)
            if (sampleRate < 8000 || sampleRate > 192000)
            {
                Logger.Log($"Invalid sample rate: {sampleRate}");
                return false;
            }
            
            // Validate bits per sample
            if (bitsPerSample != 8 && bitsPerSample != 16 && 
                bitsPerSample != 24 && bitsPerSample != 32)
            {
                Logger.Log($"Invalid bits per sample: {bitsPerSample}");
                return false;
            }
            
            // Validate calculated fields match
            uint expectedByteRate = sampleRate * numChannels * (bitsPerSample / 8);
            if (byteRate != expectedByteRate)
            {
                Logger.Log($"ByteRate mismatch: {byteRate} != {expectedByteRate}");
                return false;
            }
            
            ushort expectedBlockAlign = (ushort)(numChannels * (bitsPerSample / 8));
            if (blockAlign != expectedBlockAlign)
            {
                Logger.Log($"BlockAlign mismatch: {blockAlign} != {expectedBlockAlign}");
                return false;
            }
            
            // Skip remaining fmt data
            long fmtRemaining = chunkSize - 16;
            if (fmtRemaining > 0)
                stream.Seek(fmtRemaining, SeekOrigin.Current);
        }
        else if (chunkId == "data")
        {
            foundData = true;
            
            // Validate data chunk size is reasonable
            if (chunkSize > 50 * 1024 * 1024) // 50 MB of audio data
            {
                Logger.Log($"Data chunk too large: {chunkSize}");
                return false;
            }
            
            // Skip data
            stream.Seek(chunkSize, SeekOrigin.Current);
        }
        else
        {
            // Unknown chunk - skip it
            // But log if it's suspiciously large
            if (chunkSize > 1024 * 1024) // 1 MB metadata?
            {
                Logger.Log($"Large unknown chunk: {chunkId} ({chunkSize} bytes)");
            }
            stream.Seek(chunkSize, SeekOrigin.Current);
        }
        
        // Word-align
        if (chunkSize % 2 == 1 && stream.Position < stream.Length)
            stream.Seek(1, SeekOrigin.Current);
    }
    
    // Must have both fmt and data
    if (!foundFmt || !foundData)
    {
        Logger.Log("WAV missing required chunks");
        return false;
    }
    
    return true;
}

public static void Play(string? path)
{
    // ... existing checks ...
    
    // Validate WAV structure
    if (!ValidateWavFile(path))
    {
        Logger.Log("Invalid WAV file structure");
        return;
    }
    
    // Now safe(r) to pass to NAudio
    var reader = new AudioFileReader(path);
    // ...
}
```

**This prevents:**
- Malformed `fmt` chunk (validates all fields)
- Integer overflow in calculated sizes
- Compressed formats (rejects non-PCM)
- Excessive channels (limits to 8)
- Unreasonable sample rates
- Large metadata chunks
- Missing required chunks

**Is it worth patching?**

**YES, because:**
1. **Historical precedent** (CVE-2018-0802, CVE-2006-2916, etc.)
2. **NAudio is native code** (wraps WinMM or similar)
3. **Defense in depth** (even if rare, validation is cheap)
4. **Theme sounds execute automatically** (user doesn't click "play")

**Low likelihood, but HIGH impact if exploited.**
