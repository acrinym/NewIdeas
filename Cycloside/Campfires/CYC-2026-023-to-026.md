## CYC-2026-023: Base64-Encoded Payloads in Image/Cursor Fields

**Severity:** HIGH  
**Component:** ThemeManager, SkinManager, cursor/image loading  
**Status:** 🔍 **INVESTIGATING**

### The Attack

Theme manifests and AXAML files expect file paths for images/cursors:
```json
{
    "cursor": "cursors/pointer.cur",
    "icon": "icons/app.ico"
}
```

**But what if you provide base64-encoded data instead of a path?**

```json
{
    "cursor": "data:image/x-icon;base64,AAA...malicious_payload...AAA",
    "icon": "data:image/png;base64,iVBORw0KG...embedded_code...=="
}
```

### Why This Could Work

Many parsers support **data URIs** (RFC 2397):
- Web browsers: `<img src="data:image/png;base64,...">`
- CSS: `background-image: url(data:image/...)`
- Some desktop frameworks (Electron, WebView2, etc.)

**If Avalonia or Cycloside's file loaders support data URIs:**
1. Validator checks for valid file path (passes if checking `File.Exists`)
2. Data URI is base64-decoded at load time
3. Decoded bytes contain malicious payload
4. Payload is processed as if it were a legitimate file

### Exploitation Scenarios

#### Scenario A: Embedded Script Execution

```json
{
    "cursor": "data:text/html;base64,PHNjcmlwdD5hbGVydCgncHduZWQnKTwvc2NyaXB0Pg=="
}
```

Base64 decodes to: `<script>alert('pwned')</script>`

If cursor loader passes data to WebView or similar, script executes.

#### Scenario B: Image Parser Exploits

```json
{
    "icon": "data:image/png;base64,<crafted_PNG_with_buffer_overflow>"
}
```

Malformed PNG triggers buffer overflow in image decoder (libpng, Skia, etc.)

#### Scenario C: Format Confusion

```json
{
    "cursor": "data:audio/wav;base64,UklGRi4uLlBheWxvYWQ="
}
```

Cursor loader expects ICO/CUR format.  
Data URI provides WAV format.  
Parser confusion → crash or exploitation.

### Current Code Risk Assessment

**Need to check:**
- Does Avalonia support data URIs in image sources?
- Does `BitmapIcon`, `Cursor`, or similar accept data URIs?
- Does any theme/skin loading code use WebView2 or similar that would process data URIs?
- Are data URIs passed through file validators?

### Mitigation

```csharp
private static bool IsDataUri(string path)
{
    return path.StartsWith("data:", StringComparison.OrdinalIgnoreCase);
}

private static bool ValidateAssetPath(string path)
{
    // Reject data URIs entirely
    if (IsDataUri(path))
    {
        Logger.Log("Rejected data URI in asset path");
        return false;
    }
    
    // Then do normal path validation
    return IsValidFilePath(path);
}
```

Apply to all image, cursor, icon, and audio path validations.

## CYC-2026-024: CUR/ICO Files as RIFF Containers (Format Exploitation)

**Severity:** HIGH  
**Component:** Cursor loading, AudioService (if they share parsing code)  
**Status:** 🔍 **INVESTIGATING**

### Background: CUR/ICO File Format

**CUR (cursor) and ICO (icon) files use a RIFF-like structure:**

```
CUR/ICO Header:
  - Reserved (2 bytes): 0x0000
  - Type (2 bytes): 0x0002 for CUR, 0x0001 for ICO
  - Count (2 bytes): Number of images
  
Image Directory Entries (16 bytes each):
  - Width, Height, ColorCount, Reserved
  - Hotspot X/Y (for CUR) or Planes/BitCount (for ICO)
  - Size, Offset
  
Image Data:
  - Can be BMP, PNG, or other formats
```

**RIFF (Resource Interchange File Format) is used by WAV:**

```
RIFF Header:
  - ChunkID: "RIFF" (4 bytes)
  - ChunkSize: File size - 8
  - Format: "WAVE" (4 bytes)
  
Subchunks:
  - fmt: Format data
  - data: Audio data
```

**Both use chunked binary formats with size fields.**

### The Attack: Format Confusion

#### Attack Vector 1: CUR File with WAV Structure

Create a file named `cursor.cur` but with WAV RIFF structure inside:

```
Offset  Data         Meaning
------  -----------  -----------------------------------
0x00    00 00        Reserved (looks like ICO/CUR)
0x02    02 00        Type = CUR
0x04    01 00        Count = 1 image
0x06    [ICO entry]  Fake directory entry
0x16    RIFF         RIFF chunk starts here (offset from entry)
0x1A    <size>       Malicious size field
0x1E    WAVE         Format identifier
0x22    [chunks]     Malformed chunks, size bombs, etc.
```

**What happens:**
- Windows cursor loader sees valid CUR header (loads cursor successfully)
- Cycloside/Avalonia loads cursor (expecting ICO/CUR format)
- **If any code path tries to "be helpful" and detect format by content** rather than extension, it might parse as WAV
- WAV parser processes malformed chunks → crash or exploitation

#### Attack Vector 2: WAV File with ICO/CUR Header

Inverse attack: `sound.wav` with ICO/CUR header:

```
Offset  Data         Meaning
------  -----------  -----------------------------------
0x00    00 00        Reserved
0x02    01 00        Type = ICO
0x04    01 00        Count = 1
0x06    [entry]      Directory entry pointing past end of "ICO"
0x16    RIFF         Real WAV data starts
0x1A    ...          Actual audio
```

**If audio loader accidentally passes this to image parser first**, image parser might crash or mishandle RIFF chunks.

#### Attack Vector 3: RIFF Bomb in Cursor File

ICO/CUR format allows embedded PNG or BMP data. **What if you embed a malformed WAV RIFF structure as the "image data"?**

```
CUR Header (6 bytes)
ICO Directory Entry (16 bytes)
  → Offset points to:
RIFF Header claiming to be PNG but actually:
  - Nested RIFF chunks
  - Malicious size fields
  - Billion laughs-style chunk expansion
```

**Cursor loader tries to decode "PNG"**  
**Decoder encounters RIFF structure**  
**Confusion, crash, or exploitation**

### Why This Works

**Format auto-detection is dangerous:**
- Many libraries (libpng, libjpeg, Skia) inspect file magic bytes
- If a file claims to be one format but contains another, parsers can get confused
- Size fields in binary formats are trusted by parsers
- Malicious size fields → buffer overflows, OOM, or crashes

### Current Code at Risk

**Need to audit:**
- How does Avalonia load cursors? (`Cursor.Parse`, `new Cursor(stream)`?)
- Does it validate ICO/CUR structure before parsing?
- Does it pass cursor data through multiple parsers (BMP decoder, PNG decoder)?
- Could cursor data reach audio parsing code? (Unlikely but check EventBus, plugin interactions)

**AudioService (CYC-2026-006) already has size check**, but:
- Does it validate WAV structure?
- Does it reject non-WAV RIFF files?
- Could a malformed RIFF confuse NAudio?

### Exploitation: Polyglot Files

**A single file that is simultaneously valid CUR and valid WAV:**

```
[CUR header]
[ICO entry pointing to offset 0x20]
[Padding/garbage to reach 0x20]
RIFF [WAV structure]
```

- **As cursor:** Windows loads it fine (uses CUR header, ignores RIFF)
- **As audio:** NAudio loads it fine (skips to RIFF, ignores CUR header)
- **In Cycloside:** Could be used in both contexts, bypassing format-specific validators

### Mitigation

```csharp
private static bool ValidateCursorFile(string path)
{
    var bytes = File.ReadAllBytes(path);
    
    // Must be at least 6 bytes (CUR/ICO header)
    if (bytes.Length < 6) return false;
    
    // Check magic bytes
    if (bytes[0] != 0x00 || bytes[1] != 0x00) return false;
    
    ushort type = BitConverter.ToUInt16(bytes, 2);
    if (type != 0x0001 && type != 0x0002) return false; // ICO or CUR only
    
    // Reject if RIFF structure is detected anywhere in file
    for (int i = 0; i < bytes.Length - 4; i++)
    {
        if (bytes[i] == 'R' && bytes[i+1] == 'I' && 
            bytes[i+2] == 'F' && bytes[i+3] == 'F')
        {
            Logger.Log("Rejected cursor file containing RIFF structure");
            return false;
        }
    }
    
    return true;
}

private static bool ValidateAudioFile(string path)
{
    var bytes = File.ReadAllBytes(path);
    
    // Must start with RIFF header
    if (bytes.Length < 12) return false;
    if (bytes[0] != 'R' || bytes[1] != 'I' || 
        bytes[2] != 'F' || bytes[3] != 'F') return false;
    
    // Must be WAVE format
    if (bytes[8] != 'W' || bytes[9] != 'A' || 
        bytes[10] != 'V' || bytes[11] != 'E') return false;
    
    // Reject if ICO/CUR header is detected
    if (bytes.Length > 6 && bytes[0] == 0x00 && bytes[1] == 0x00)
    {
        ushort possibleType = BitConverter.ToUInt16(bytes, 2);
        if (possibleType == 0x0001 || possibleType == 0x0002)
        {
            Logger.Log("Rejected audio file containing ICO/CUR structure");
            return false;
        }
    }
    
    return true;
}
```

## CYC-2026-025: Parser Confusion - CUR Parser Reading WAV

**Severity:** MEDIUM  
**Component:** Cursor loading pipeline  
**Status:** 🔍 **INVESTIGATING**

### The Attack

If any code in the cursor loading pipeline uses **content-based format detection** rather than trusting file extensions:

```csharp
// Dangerous pattern:
var stream = File.OpenRead(cursorPath);
var image = Image.Load(stream); // Image.Load auto-detects format
```

**Attacker provides `evil.cur` containing WAV RIFF structure.**

**Image loader tries every decoder:**
- PNG decoder (fails)
- BMP decoder (fails)
- GIF decoder (fails)
- ...eventually tries some generic "RIFF container" handler
- Treats WAV chunks as image data
- **Crash, exploit, or undefined behavior**

### Why This Matters

**Avalonia uses SkiaSharp for image decoding.**  
**SkiaSharp uses libpng, libjpeg, libwebp, etc.**  
**These are native libraries with historical buffer overflow issues.**

**A file crafted to exploit a specific decoder's vulnerability**, disguised as a cursor file, could:
- Trigger buffer overflow
- Cause arbitrary code execution
- Bypass AXAML/theme validation entirely (CUR files aren't validated as AXAML)

### Mitigation

**Never use auto-detection for theme assets:**

```csharp
private static Cursor LoadCursorSafe(string path)
{
    // Validate format BEFORE passing to parser
    if (!ValidateCursorFile(path))
        throw new InvalidDataException("Invalid cursor file");
    
    // Use explicit format, not auto-detection
    using var stream = File.OpenRead(path);
    return new Cursor(stream); // Avalonia's cursor parser
}
```

**Reject files that don't match expected format exactly.**

## CYC-2026-026: RIFF Chunk Size Integer Overflow

**Severity:** HIGH  
**Component:** AudioService (NAudio), any RIFF parsers  
**Status:** 🔍 **INVESTIGATING**

### The Attack

RIFF format uses 32-bit unsigned integers for chunk sizes:

```
RIFF Header:
  ChunkID: "RIFF" (4 bytes)
  ChunkSize: 0xFFFFFFFF (4 bytes) ← Malicious value
  Format: "WAVE"
  
Subchunk:
  SubchunkID: "data"
  SubchunkSize: 0xFFFFFFFE (4 bytes) ← Another large value
  Data: [actual bytes]
```

**Parser logic:**
```csharp
uint chunkSize = ReadUInt32(stream);
byte[] buffer = new byte[chunkSize]; // Integer overflow if chunkSize > int.MaxValue
stream.Read(buffer, 0, buffer.Length);
```

**If `chunkSize` is 0xFFFFFFFF (4,294,967,295 bytes = 4GB):**
- 32-bit process: Allocation fails or wraps to negative
- 64-bit process: Might allocate 4GB (if physical RAM available)
- **Either way: Denial of service**

### Current Risk in AudioService

**From CYC-2026-006 mitigation:**
```csharp
private static readonly long MaxAudioFileSize = 10 * 1024 * 1024; // 10 MB

var fileInfo = new FileInfo(path);
if (fileInfo.Length > MaxAudioFileSize) return;
```

**This checks FILE SIZE on disk, not CHUNK SIZES inside the file.**

**A malicious WAV could be:**
- 100 KB on disk (passes size check)
- Claims to have 4 GB of audio data via chunk size field
- NAudio attempts to allocate 4 GB buffer
- **OutOfMemoryException**

### Exploitation

**Craft a WAV file:**
```
RIFF [ChunkSize = 0xFFFFFFFF]
WAVE
fmt [ChunkSize = 16, valid format data]
data [ChunkSize = 0xFFFFFFFF, actual data = 1 KB]
```

**File size on disk: 1 KB**  
**NAudio sees:** "This file has 4 GB of audio data"  
**NAudio allocates:** 4 GB buffer  
**Result:** Crash

### Mitigation

**Validate RIFF structure before passing to NAudio:**

```csharp
private static bool ValidateWavStructure(string path)
{
    using var stream = File.OpenRead(path);
    using var reader = new BinaryReader(stream);
    
    // Read RIFF header
    var riff = reader.ReadChars(4);
    if (new string(riff) != "RIFF") return false;
    
    uint fileSize = reader.ReadUInt32();
    
    // File size in header must match actual file size (within reason)
    if (fileSize > stream.Length || fileSize < stream.Length - 8)
    {
        Logger.Log("WAV file size mismatch (possible integer overflow attack)");
        return false;
    }
    
    var wave = reader.ReadChars(4);
    if (new string(wave) != "WAVE") return false;
    
    // Validate each subchunk
    while (stream.Position < stream.Length - 8)
    {
        var chunkId = reader.ReadChars(4);
        uint chunkSize = reader.ReadUInt32();
        
        // Chunk size must not exceed remaining file size
        long remaining = stream.Length - stream.Position;
        if (chunkSize > remaining)
        {
            Logger.Log($"Invalid chunk size ({chunkSize} > {remaining})");
            return false;
        }
        
        // Chunk size must be reasonable (< 50 MB per chunk)
        if (chunkSize > 50 * 1024 * 1024)
        {
            Logger.Log($"Chunk size too large ({chunkSize} bytes)");
            return false;
        }
        
        // Skip chunk data
        stream.Seek(chunkSize, SeekOrigin.Current);
        
        // Chunks are word-aligned (skip padding byte if needed)
        if (chunkSize % 2 == 1 && stream.Position < stream.Length)
            stream.Seek(1, SeekOrigin.Current);
    }
    
    return true;
}

public static void Play(string? path)
{
    // ... existing checks ...
    
    if (!ValidateWavStructure(path))
    {
        Logger.Log("Invalid WAV structure");
        return;
    }
    
    // Now safe to pass to NAudio
    var reader = new AudioFileReader(path);
    // ...
}
```
